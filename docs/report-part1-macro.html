Chapter 1. 프로젝트 개요

1.1 배경 및 동기

한국 주식시장(KRX)은 KOSPI 약 950종목, KOSDAQ 약 1,780종목을 포함하여 총 2,733개 상장 종목을 보유한 아시아 주요 자본시장이다. 개인 투자자("개미")의 시장 참여율이 세계적으로 높은 한국 시장에서 기술적 분석(Technical Analysis)은 매매 의사결정의 핵심 도구로 활용된다.

기존 HTS(Home Trading System)는 데스크탑 전용 설치형 소프트웨어로, 브라우저 기반 접근이 제한되며 패턴 인식 기능이 제한적이거나 수동 판별에 의존한다. 키움증권 영웅문, 삼성증권 POP, NH투자증권 QV 등 주요 HTS는 캔들스틱 패턴을 자동 감지하는 기능을 제공하지 않거나, 제공하더라도 5~8종에 한정된다.

CheeseStock은 이러한 한계를 극복하기 위해 설계된 웹 기반 플랫폼으로, 26종 캔들스틱/차트 패턴 자동 감지, 16종 기술적 지표 시그널, WLS 회귀 기반 수익률 예측을 하나의 브라우저 인터페이스에 통합한다.

1.2 효율적 시장 가설(EMH) vs 기술적 분석

Fama(1970)의 효율적 시장 가설(Efficient Market Hypothesis)은 과거 가격 정보가 즉시 반영되어 기술적 분석의 초과 수익 가능성을 부정한다. 그러나 행동재무학(Behavioral Finance) 관점에서 투자자의 비합리적 행동(군집행동, 과잉 반응, 손실 회피)이 가격 패턴을 형성한다는 실증 연구가 축적되었다 (Lo, Mamaysky & Wang, 2000; Bulkowski, 2005).

CheeseStock은 패턴 감지 결과의 과거 수익률 통계를 제공함으로써, 사용자가 패턴의 통계적 유의성을 직접 검증할 수 있도록 설계되었다. 백테스트 모듈(backtester.js, 665줄)은 각 패턴 발생 후 1/3/5/10/20일 수익률, 승률, 95% 신뢰구간을 산출하며, WLS(Weighted Least Squares) 회귀를 통한 예측 수익률도 제공한다.

본 시스템은 패턴 감지 및 통계 정보 제공을 목적으로 하며, 투자 권유가 아님을 명시한다. 모든 수익률 표기에는 "과거 수익률은 미래 수익을 보장하지 않습니다" 면책 조항이 포함된다 (index.html L389).

1.3 설계 목표

  1. Zero-Build Architecture — 번들러/트랜스파일러 없이 <script defer>로 16개 모듈을 순서 실행. 개발 환경 설정 비용 제거.
  2. Dual-Mode Reliability — Kiwoom WS 실시간 + Cloudflare 파일 정적 모드. 서버 장애 시에도 전일 종가 기준 차트 제공.
  3. Main-Thread Offloading — 패턴/시그널/백테스트 연산을 Web Worker로 격리. 차트 렌더링은 60fps 유지.
  4. Korean Market Specifics — 상승=빨강/하락=파랑 색상 체계, KRX 거래 시간(09:00-15:30 KST), 가격제한폭(±30%) 검증, 호가단위 대응.
  5. Production Security — Kiwoom 계정 잠금 방지 7-Layer 로그인 보호, SRI 무결성 검증, localStorage 안전 접근.

2.3 defer 로딩과 순서 보장

HTML5의 defer 속성은 스크립트를 비동기로 다운로드하되, DOM 파싱 완료 후 <script> 태그 등장 순서대로 실행한다. CheeseStock은 이 메커니즘을 활용하여 빌드 시스템 없이도 모듈 간 의존성 순서를 보장한다. 

<!-- index.html L611-L626: 16개 defer 스크립트 (순서 엄수) -->
<script defer src="js/colors.js"></script>           <!-- 1. 색상 상수 -->
<script defer src="js/data.js"></script>             <!-- 2. 시드 데이터 -->
<script defer src="js/api.js"></script>              <!-- 3. 데이터 서비스 -->
<script defer src="js/realtimeProvider.js"></script>  <!-- 4. WS 클라이언트 -->
<script defer src="js/indicators.js"></script>       <!-- 5. 기술적 지표 -->
<script defer src="js/patterns.js"></script>         <!-- 6. 패턴 엔진 -->
<script defer src="js/signalEngine.js"></script>     <!-- 7. 시그널 엔진 -->
<script defer src="js/chart.js"></script>            <!-- 8. 차트 관리자 -->
<script defer src="js/patternRenderer.js"></script>  <!-- 9. 패턴 렌더러 -->
<script defer src="js/signalRenderer.js"></script>   <!-- 10. 시그널 렌더러 -->
<script defer src="js/backtester.js"></script>       <!-- 11. 백테스터 -->
<script defer src="js/sidebar.js"></script>          <!-- 12. 사이드바 -->
<script defer src="js/patternPanel.js"></script>     <!-- 13. 패턴 패널 -->
<script defer src="js/financials.js"></script>       <!-- 14. 재무 지표 -->
<script defer src="js/drawingTools.js"></script>     <!-- 15. 드로잉 도구 -->
<script defer src="js/app.js"></script>              <!-- 16. 앱 오케스트레이터 -->
Critical Constraint: 이 순서를 변경하면 전역 변수 참조 오류(ReferenceError)가 발생한다. 예를 들어 patterns.js가 indicators.js보다 먼저 로드되면 calcATR 함수를 찾을 수 없어 패턴 엔진이 실패한다.

3.3 Web Worker 격리 경계

analysisWorker.js는 별도의 JavaScript 실행 컨텍스트에서 동작한다. 메인 스레드의 전역 변수에 접근할 수 없으며, importScripts()로 필요한 모듈만 Worker 스코프에 로드한다. 

// analysisWorker.js L42-L49
importScripts(
  'colors.js',        // KRX_COLORS (패턴 품질 평가에 사용)
  'indicators.js',    // calcATR, calcMA, calcEMA ...
  'patterns.js',      // patternEngine.analyze()
  'signalEngine.js',  // signalEngine.analyze()
  'backtester.js'     // backtester.backtestAll()
);

중요한 제약사항: IndicatorCache 객체는 내부에 클로저와 함수 참조를 포함하므로 Structured Clone Algorithm에 의해 직렬화할 수 없다. 따라서 Worker에서 생성된 cache 객체는 postMessage()로 전달하지 않고, signals와 stats만 전달한다 (analysisWorker.js L98-L105).

4.4 데이터 검증 (Sanitization)

모든 캔들 데이터는 _sanitizeCandles() (api.js L494-L530)에서 3단계 검증을 거친다.

  1. OHLCV 유효성 — high ≥ low, open/close ∈ [low, high], volume ≥ 0, 가격 > 0 (_validateCandle() L464-L486)
  2. 가격제한폭 검증 — 전일 대비 ±35% 초과 시 경고 후 제거 (KRX 규정 ±30% + 5% 여유)
  3. 메모리 상한 — 일봉 최대 MAX_CANDLES_DAILY = 2000건, 분봉 최대 MAX_CANDLES_INTRADAY = 500건 (api.js L181-L183)

5.3 비동기 WS 프로브 (Non-Blocking)

초기 로딩 시 WS 서버 가용성을 확인하는 프로브는 비동기로 실행된다 (api.js L200-L213). 프로브 실행 동안 mode를 'file'로 즉시 전환하여 차트를 먼저 표시하고, 프로브 성공 시 백그라운드에서 'ws'로 복원한다. 

// api.js L200-L213: 비동기 WS 프로브 — 초기 로딩 3초 블로킹 제거
KRX_API_CONFIG.mode = 'file';   // 즉시 file 전환
this.probeWsServer(KRX_API_CONFIG.wsUrl, 3000).then(function(ok) {

// api.js L367-L370: 데이터 출처 태그

6.2 L1: 메모리 캐시

KRXDataService.cache 객체(api.js L190)는 {'code-timeframe': { candles, lastUpdate }} 형태의 해시맵이다. 최대 MAX_CACHE_ENTRIES = 50개 항목을 보유하며, 초과 시 _pruneCache()(L536-L549)가 LRU 방식으로 가장 오래된 항목부터 삭제한다. 

// api.js L283-L287: L1 캐시 확인 (getCandles 진입점)
const cached = this.cache[key];
const TTL = timeframe === '1d' ? 3600000 : 300000;  // 일봉 1시간, 분봉 5분
if (cached && (Date.now() - cached.lastUpdate) < TTL)
  return cached.candles;

6.3 L2: IndexedDB 캐시

_idb 객체(api.js L23-L72)는 IndexedDB를 래핑하여 페이지 리로드 후에도 일봉 데이터를 즉시 제공한다. 데이터베이스명은 'krx_cache', 오브젝트 스토어명은 'candles', 키는 "종목코드-타임프레임" 형식이다.

  • fail-silent 설계: 모든 연산은 async이며 예외 발생 시 null 반환. IndexedDB 미지원 환경(일부 프라이빗 모드)에서도 앱이 정상 동작한다.
  • 일봉 전용: 분봉 데이터는 실시간성이 중요하므로 L2 캐시에 저장하지 않는다.
  • 24시간 TTL: L2 데이터가 24시간 이내면 L3 네트워크 요청을 건너뛴다 (L295).

6.4 L3: 네트워크 소스

L1, L2 캐시가 모두 미스인 경우 KRX_API_CONFIG.mode에 따라 적절한 L3 소스에서 데이터를 가져온다 (api.js L311-L357). 성공 시 결과를 L1 캐시에 저장하고, 일봉인 경우 L2 IndexedDB에도 비동기 기록한다 (fire-and-forget, L376-L379).

6.5 Worker 분석 캐시

analysisWorker.js L32-L38에 정의된 _analyzeCache는 동일한 캔들 배열에 대한 중복 분석을 방지한다. 캐시 키는 key = candles.length + '_' + last.time + '_' + last.open + '_' + last.close 으로 구성되며, 드래그 이벤트에서 동일 visible 구간이 반복 요청될 때 캐시 적중률이 높다.

Chapter 7. Web Worker 분석 파이프라인

7.1 아키텍처 개요

패턴 감지, 시그널 분석, 백테스트는 O(n) 이상의 연산 복잡도를 가지며, 2,000 캔들 기준으로 수십 밀리초가 소요된다. 이를 메인 스레드에서 실행하면 차트 렌더링이 차단되어 UI jank가 발생하므로, Web Worker로 오프로드한다.

7.2 메시지 프로토콜

Table 1-9. Worker 메시지 프로토콜
방향type페이로드설명

7.3 Version-Based Stale 결과 거부

사용자가 종목 A의 분석 결과가 도착하기 전에 종목 B로 전환하면, 종목 A의 결과는 더 이상 유효하지 않다. CheeseStock은 version 필드를 통해 이 문제를 해결한다. 

// app.js L311: 전역 Worker version 카운터
let _workerVersion = 0;

// 종목 변경 시 version 증가 (app.js L2163)
_workerVersion++;

// 분석 요청 시 현재 version 전송 (app.js L1823-L1828)
_analysisWorker.postMessage({
  type: 'analyze',
  candles: candles,
  realtimeMode: _realtimeMode,
  version: _workerVersion,    // ← 현재 version 전달
});

// 결과 수신 시 version 비교 (app.js L1434)
if (msg.version !== _workerVersion) return;  // stale → 무시

동일한 패턴이 드래그 분석에도 적용된다: _dragVersion (app.js L314)은 드래그 시마다 증가하며, 이전 드래그의 결과가 뒤늦게 도착하면 무시된다.

7.4 3초 스로틀

실시간 모드에서 매 틱마다 분석을 실행하면 불필요한 CPU 소비가 발생한다. app.js L110에 정의된 _lastPatternAnalysis 타임스탬프를 통해 최소 3초 간격으로 분석을 제한한다. 

// app.js L1664-L1685: 3초 스로틀
if (patternEnabled && now - _lastPatternAnalysis > 3000) {
  if (_analysisWorker && _workerReady) {
    _requestWorkerAnalysis();        // Worker 비동기
  } else {
    // 폴백: 메인 스레드 동기 분석
    detectedPatterns = patternEngine.analyze(candles);
  }
  _lastPatternAnalysis = now;
}

7.5 드래그 분석과 150ms 디바운스

차트를 드래그하면 visible range가 변경되며, 해당 구간의 패턴을 즉시 분석한다. 빠른 드래그 시 불필요한 분석을 방지하기 위해 150ms 디바운스를 적용한다 (app.js L948-L969). 

// app.js L948-L969: 드래그 분석 디바운스
if (_dragDebounceTimer) clearTimeout(_dragDebounceTimer);

_dragDebounceTimer = setTimeout(() => {
  _dragVersion++;
  _dragClampFrom = clampFrom;
  _analysisWorker.postMessage({
    type: 'analyze',
    candles: visibleCandles,
    realtimeMode: false,
    version: _dragVersion,
    source: 'drag',           // ← 드래그 결과 구분자
  });
}, 150);

7.6 Worker 장애 복구

Worker importScripts 실패 시 type: 'error' 메시지가 전송된다 (analysisWorker.js L52-L55). 메인 스레드에서는 _workerRestartCount(최대 3회, app.js L318)를 추적하여 Worker 재생성을 시도하며, 3회 실패 시 메인 스레드 동기 분석으로 폴백한다.

7.7 백테스트 캐시 Pre-seeding

'backtest' 요청 시, Worker는 이전 'analyze' 요청에서 이미 계산한 patternEngine.analyze() 결과를 재사용한다 (analysisWorker.js L142-L148). 이를 통해 O(n) 패턴 분석의 중복 실행을 방지한다. 

// analysisWorker.js L142-L148: 백테스트 캐시 pre-seeding
const btCacheKey = _makeCacheKey(candles);
if (_analyzeCache.key === btCacheKey && _analyzeCache.patterns) {
  backtester._analyzeCache = {
    _candles: candles,
    patterns: _analyzeCache.patterns,  // 이전 analyze 결과 재사용
  };
}

8.1 개요

sw.js(138줄)는 Progressive Web App(PWA)의 핵심 구성 요소로, 3가지 캐싱 전략을 요청 유형에 따라 선택 적용한다. 캐시 이름은 'cheesestock-v2'이며, 버전 변경 시 이전 캐시가 자동 정리된다.

8.2 정적 자산 목록

sw.js L11-L33에 정의된 STATIC_ASSETS 배열은 오프라인 실행에 필요한 20개 자산을 포함한다:

  • /, /index.html, /favicon.svg
  • /css/style.css
  • 16개 JS 모듈 + /js/analysisWorker.js + /js/app.js

8.4 전략 선택 근거

Cache-First (정적 자산)

HTML, CSS, JS 파일은 배포 시에만 변경되므로 캐시 우선이 적합하다. 배포 시 CACHE_NAME의 버전을 올려('cheesestock-v2' → 'v3') 이전 캐시를 무효화한다. activate 이벤트에서 이전 버전 캐시를 삭제한다 (sw.js L54-L66).

Network-First (데이터 파일)

data/ 하위의 JSON 파일은 매일 갱신되므로 최신 데이터를 우선한다. 네트워크 실패 시에만 캐시된 이전 데이터를 사용하여 오프라인 지원을 제공한다. 이 설계는 "stale data is worse than no data" 원칙에 따르되, 오프라인 환경에서는 어제 데이터라도 표시하는 것이 빈 차트보다 유용하다는 판단이다.

Stale-While-Revalidate (외부 CDN)

TradingView Lightweight Charts, Pretendard 폰트, JetBrains Mono 폰트 등 외부 CDN 자원은 변경 빈도가 낮지만 완전히 정적이지 않다. 캐시된 버전을 즉시 제공하여 로딩 속도를 확보하면서, 백그라운드에서 최신 버전을 가져와 캐시를 갱신한다.

8.5 비-인터셉트 대상

  • WebSocket 요청 (ws:, wss:): Service Worker가 기술적으로 인터셉트할 수 없다 (sw.js L73).
  • Non-GET 요청: 캐싱 불가 — 네트워크로 직접 전달 (L76).

10.3 SRI (Subresource Integrity)

외부 CDN에서 로드하는 TradingView Lightweight Charts 라이브러리에 SRI 해시를 적용하여 CDN 공급망 공격(supply-chain attack)을 방지한다. 

<!-- index.html L24-L26 -->
<script defer
  src="https://unpkg.com/lightweight-charts@4.2.3/dist/
       lightweight-charts.standalone.production.js"
  integrity="sha384-stKllnUqA9AD0gsKCuUtf5XlqAW7PwIg
             DagoNsTWkjkBmJ/GZ/uHTgEBxdLV2VSK"
  crossorigin="anonymous">
</script>

SHA-384 해시가 일치하지 않으면 브라우저가 스크립트 실행을 거부하므로, unpkg CDN이 해킹되어도 악성 코드가 실행되지 않는다. crossorigin="anonymous"는 SRI 검증의 전제 조건이다.

10.4 localStorage 안전 접근

프라이빗 브라우징, 스토리지 용량 초과, iframe 제한 등의 환경에서 localStorage 접근이 예외를 발생시킬 수 있다. CheeseStock은 모든 localStorage 접근을 try/catch로 감싸고, 실패 시 기본값으로 폴백한다. 

// app.js L472-L480: 안전한 localStorage 읽기
function _loadPrefs() {
  try {
    var raw = localStorage.getItem(_PREFS_KEY);
    if (!raw) return null;
    return JSON.parse(raw);
  } catch (e) {
    return null;     // localStorage 접근 불가 — 기본 설정 사용
  }
}

// app.js L482-L490: 안전한 localStorage 쓰기
function _savePrefs(partial) {
  try {
    var prev = _loadPrefs() || {};
    var merged = Object.assign(prev, partial);
    localStorage.setItem(_PREFS_KEY, JSON.stringify(merged));
  } catch (e) {
    // localStorage 비활성 또는 용량 초과 — 무시
  }
}

10.5 도메인 기반 WS URL 보호

사용자가 localStorage에 ws://localhost:8765를 저장한 상태에서 프로덕션 도메인(cheesestock.co.kr)에 접속하면, 로컬 WS 주소가 무의미하게 적용되어 연결 실패가 발생할 수 있다. 이를 방지하기 위해 api.js L89-L101에서 프로덕션 도메인 + localStorage localhost URL 조합을 감지하여 무시한다. 

// api.js L89-L101: 프로덕션 도메인에서 localhost URL 무시
var _savedPrefs = JSON.parse(localStorage.getItem('krx-prefs'));
if (_savedPrefs && _savedPrefs.wsUrl) {
  var _isLocalSaved = _savedPrefs.wsUrl.includes('localhost')
                   || _savedPrefs.wsUrl.includes('127.0.0.1');
  var _isProdDomain = _defaultWsUrl.startsWith('wss://');
  if (!(_isLocalSaved && _isProdDomain)) {
    KRX_API_CONFIG.wsUrl = _savedPrefs.wsUrl;
  }
}

10.6 WebSocket 재연결 보호

realtimeProvider.js의 WS 재연결 로직은 지수 백오프(exponential backoff)와 30% 지터를 적용하여 다수 클라이언트의 동시 재연결(thundering herd)을 방지한다. 

// realtimeProvider.js L162-L175: 지수 백오프 + 지터
var baseDelay = this._wsReconnectDelay     // 초기 2000ms
              * Math.pow(1.5, this._wsReconnectCount - 1);
var jitter = Math.random() * baseDelay * 0.3;  // 30% 지터
var delay = Math.min(baseDelay + jitter, 15000);  // 최대 15초

// 최대 재연결 시도: 20회 (~5분 커버리지)
if (this._wsReconnectCount >= this._wsMaxReconnect) {
  this._emit({ error: true, message: 'WebSocket 재연결 실패' });
}

재연결 시도 횟수는 _wsReconnectCount로 추적되며, _wsMaxReconnect = 20회 초과 시 연결을 포기하고 사용자에게 에러를 표시한다. 성공적으로 연결되면 카운터가 0으로 리셋된다 (L117).
