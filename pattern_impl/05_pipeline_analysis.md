## 2. 장점 분석

### 2.1 학술적 기반의 깊이 (Depth of Academic Foundation)

**평가: ★★★★★ (탁월)**

- 24편의 학술 문서가 수학(01), 통계(02), 물리(03), 심리(04), 금융(05),
  기술적 분석(06), 알고리즘(07), 참고문헌(08), 게임이론(09), 최적제어(10),
  RL(11,11B), 극단값(12), 정보기하(13), 재무관리(14), ML(15), 패턴레퍼런스(16),
  회귀백테스팅(17), 행동미시구조(18), 소셜네트워크(19), KRX이상(20),
  적응형패턴(21), 학습상수(22), APT팩터(23), 행동계량화(24) 전 영역을 커버
- 각 공식에 원전 논문/교과서 참조가 명시되어 있음
- 수식 검증이 이루어짐 (VaR/ES, 켈리 연속형, ATR 정규화 등)
- 10단계 학습 경로가 설계되어 있어 신규 개발자의 온보딩 지원

업계 비교:
- TradingView: Pine Script 문서는 사용법 중심, 학술 근거 거의 없음
- MetaTrader: MQL5 Reference는 API 중심, 패턴 이론 미기술
- Bloomberg Terminal: 내부 리서치 팀 보유 but 비공개
- **본 프로젝트가 오픈소스 프로젝트 중 학술 기반 깊이에서 상위 1%**

### 2.2 ATR 정규화 일관성 (Normalization Consistency)

**평가: ★★★★☆ (우수)**

- 거의 모든 패턴에서 ATR 기반 임계값 사용
  - 적삼병: `body >= ATR * 0.3` (L157)
  - 삼각형: `|H1-H2| < ATR * 0.5` (L614)
  - 쐐기: ATR 정규화 기울기 비교 (L740-741)
- 가격 수준 독립적: 5,000원 주식과 500,000원 주식에 동일 기준 적용
- 07 S3.5에서 권장한 "body/ATR > 0.3" 기준이 실제 구현에 반영

개선 여지:
- `_detectTrend()` L47: `norm = slope / (sy/n * 0.01)` -- 가격 기반 정규화 (ATR이 아님)
- 권장: `norm = slope / ATR` 방식으로 통일

### 2.3 다요인 품질 점수 (Multi-Factor Quality Scoring)

**평가: ★★★★☆ (우수)**

- `_quality()` 함수로 5가지 요인 통합 (body, volume, trend, shadow, extra)
- 가중치 (0.25, 0.25, 0.20, 0.15, 0.15)가 명시되어 수정 용이
- 07 S7.1의 프레임워크와 구조적으로 일치

개선 여지:
- 가중치가 고정값: 백테스팅 기반 캘리브레이션 미적용 (07 S3.6 권장)
- 패턴별 차등 가중치 미적용 (적삼병과 도지의 품질 요인이 다름)

### 2.4 리스크 관리 통합 (Risk Management Integration)

**평가: ★★★★☆ (우수)**

- 모든 패턴에 `stopLoss`, `priceTarget` 포함
- ATR 기반 손절가 (2x ATR 기본)
- 패턴 높이 기반 목표가 (1.5x 패턴 높이)

개선 여지:
- Risk-Reward Ratio 명시적 계산/표시 없음
- 패턴별 차등 ATR 배수 미적용 (07에서 권장)
- 켈리 기준 포지션 사이징 미구현 (14에서 기술)

### 2.5 코드-이론 매핑 문서화 (Code-Theory Mapping)

**평가: ★★★☆☆ (보통)**

## 3. 단점 분석

### 3.1 학술 -> 구현 변환의 암묵적 과정

**문제: 학술 문서의 수식이 코드로 변환되는 과정이 문서화되지 않음**

```
예시:

학술 (07 S3.5):
  "적삼병: body[k] / ATR(14) > 0.3"

코드 (patterns.js L157):
  "if (b0 < a * 0.3 || b1 < a * 0.3 || b2 < a * 0.3) continue;"

변환은 정확하지만:
  - 왜 0.3인가? (Bulkowski 통계에서 유의미한 봉 크기의 하한)
  - KRX 시장에서도 0.3이 적절한가? (백테스팅 필요)
  - 이 임계값을 변경하면 어떤 영향이 있는가? (민감도 분석 없음)
```

**개선**: pattern_impl/ 문서에서 각 임계값의 근거를 명시 (이 문서가 그 역할)

### 3.2 검증 체계의 부재

**문제: 패턴 탐지의 정확성을 자동으로 검증하는 시스템이 없음**

```
현재 상태:
  - 단위 테스트: 없음
  - 백테스팅 엔진: 없음
  - 패턴 정확도 측정: 없음
  - 거짓 양성/음성 비율: 알 수 없음

대비:
  - TradingView: 내장 Strategy Tester + 커뮤니티 검증
  - MetaTrader: 내장 Strategy Tester + 최적화 도구
  - Bloomberg: 독립 리서치 팀의 백테스팅
```

**개선 방안**:
1. 프론트엔드 백테스팅 모듈 (15 S6 기반): `data/` 폴더의 실제 데이터로 워크포워드 분석
2. 콘솔 로그 기반 간이 검증: 각 종목별 패턴 발생 횟수 + 5봉 후 수익률
3. 장기적으로: Python 백테스팅 스크립트 (`scripts/backtest.py`)

### 3.3 지표-패턴 분리 (Indicator-Pattern Separation)

**상태: 해결됨 (Phase 5)**

```
해결된 아키텍처:
  indicators.js: calcRSI(), calcMACD(), calcBB() 등 9종 전역 함수 + IndicatorCache
  patterns.js: PatternEngine 클래스 -- indicators.js 전역 함수 참조 가능
  signalEngine.js: SignalEngine -- IndicatorCache + patternEngine 양쪽 참조
  chart.js: ChartManager -- 지표 함수는 indicators.js에 위임

로드 순서:
  현재: colors.js -> data.js -> api.js -> realtimeProvider.js -> indicators.js
        -> patterns.js -> signalEngine.js -> chart.js -> ...

개선 결과:
  - 지표 계산 함수가 indicators.js로 분리되어 모든 파일에서 참조 가능
  - ATR 중복 계산 해소 (indicators.js calcATR이 표준)
  - IndicatorCache로 lazy evaluation + 캐시 최적화
  - signalEngine.js가 복합 시그널 구현 (캔들+지표 동시 활용)
```

**남은 개선 사항**:
- patterns.js 내부 `_calcATR()`은 아직 독자 구현 유지 (indicators.js calcATR과 동일 로직)
- 장기: 모듈 시스템 도입 (ES Modules)

### 3.4 하드코딩된 임계값

**문제: 패턴 탐지의 핵심 임계값들이 코드 내에 하드코딩**

```
patterns.js 내 하드코딩된 임계값 목록:

L157:  body >= ATR * 0.3       (적삼병 최소 몸통)
L160:  upperWick <= body * 0.5  (적삼병 윗꼬리 한도)
L232:  lowerShadow >= body * 2  (해머 아래꼬리)
L232:  upperShadow <= body * 0.3 (해머 윗꼬리 한도)
L232:  body >= range * 0.1      (해머 최소 몸통)
L388:  body <= range * 0.05     (도지 몸통 한도)
L391:  range >= ATR * 0.3       (도지 최소 범위)
L425:  prevBody >= ATR * 0.1    (장악형 이전봉 최소)
L426:  currBody >= ATR * 0.2    (장악형 현재봉 최소)
L479:  prevBody >= ATR * 0.3    (잉태형 이전봉 최소)
L479:  currBody <= prevBody * 0.5 (잉태형 현재봉 한도)
L537:  body0 >= ATR * 0.3       (샛별형 첫째봉 최소)
L538:  body1 <= ATR * 0.2       (샛별형 별 최대)
L614:  |H1-H2| <= ATR * 0.5    (삼각형 수평선 허용)
L849:  |L1-L2| <= ATR * 1.0    (이중바닥 허용)
L930:  |LS-RS| / Head <= 0.05   (머리어깨 대칭)
```

**개선 방안**:
```javascript
// 패턴 설정 객체로 중앙 관리
const PATTERN_CONFIG = {
  threeWhiteSoldiers: {
    minBodyATR: 0.3,      // 최소 몸통 / ATR
    maxWickBody: 0.5,     // 최대 윗꼬리 / 몸통
    // ... 학술 근거: 07 S3.5, Bulkowski 통계
  },
  hammer: {
    minShadowBody: 2.0,   // 최소 아래꼬리 / 몸통
    maxUpperBody: 0.3,    // 최대 윗꼬리 / 몸통
    minBodyRange: 0.1,    // 최소 몸통 / 전체범위
    // ... 학술 근거: Nison (1991)
  },
  // ...
};
```

### 3.5 시간프레임 적응성 부재

**문제: 모든 임계값이 시간프레임 독립적으로 설정**

```
ATR 정규화가 부분적으로 해결하지만:
  - 1분봉의 노이즈 수준 vs 일봉의 구조적 패턴 차이 미고려
  - 분봉에서 도지(body < range*0.05)가 과다 감지될 가능성
  - 주봉에서 적삼병(3연속 양봉)이 과소 감지될 가능성

권장: 시간프레임별 임계값 조정 계수
  - 1분봉: 0.7x (더 관대한 임계값)
  - 5분봉: 0.85x
  - 15분봉: 0.9x
  - 일봉: 1.0x (기본)
  - 주봉: 1.2x (더 엄격한 임계값)
```

## 6. 핵심 결론

### 이 프로젝트의 차별적 강점
1. **학술적 깊이**: 16편의 체계적 이론 문서는 오픈소스 트레이딩 도구 중 유일무이
2. **ATR 정규화**: 가격 독립적 패턴 탐지로 모든 KRX 종목에 일관 적용
3. **다요인 품질 점수**: 단순 이진 탐지를 넘어선 확률적 접근
4. **리스크 통합**: 모든 시그널에 손절/목표가 포함

### 가장 시급한 개선
1. **임계값 중앙화** -> 하드코딩된 매직 넘버를 설정 객체로 추출
2. **Tier 2-3 패턴 추가** -> 42종 중 26종 구현 (61.9%), 80%+ 목표
3. **시간프레임 적응** -> 분봉에서 과다 감지되는 거짓 양성 해소

### 장기 비전
학술 -> 정의 -> 구현 -> **검증** -> 개선의 순환 파이프라인.
Phase 7에서 백테스팅 엔진을 도입하여 폐쇄 루프(closed loop)의 기반을 마련했다.
다음 단계는 backtester의 수익률 통계를 임계값 캘리브레이션에 피드백하는 것이다.
Python 백테스팅 스크립트(scripts/backtest.py)를 통한 대규모 검증도 병행 추진.
